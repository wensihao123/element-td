# 元素反应塔防（暂定名）— 项目说明

基于 Godot 4.x 的塔防游戏。核心创新点：炮塔为敌人附着元素状态，不同元素状态之间发生**反应**，产生额外战术效果。本文档是项目的设计与架构基准，Claude Code session 开发时以此为准。

---

## 1. 设计支柱（所有设计决策的最高裁判）

1. **反应 > 单体输出**：任何单元素流派堆到顶，效率必须低于合理的双元素组合。纯色流可通普通难度，高难必须玩反应。
2. **顺序即策略**：塔的摆放位置和顺序决定反应发生的地点与时机，这是核心策略深度来源。
3. **战场必须可读**：状态图标、元素量环、反应特效、飘字要极其清晰。玩家看不懂反应 = 深度变噪音。

---

## 2. 核心机制：元素附着与 Gauge 制反应

### 2.1 元素与基础状态

| 元素 | 基础状态（gauge > 0 时持续生效） |
|------|--------------------------------|
| 🔥 火 | 灼烧：持续伤害（DoT） |
| ❄️ 冰 | 减速 30% |
| ⚡ 电 | 周期性小伤害 |
| ☠️ 毒 | 腐蚀：降低护甲 |

### 2.2 Gauge（元素量）规则

- 每个敌人身上**最多挂 1 种**元素状态，以浮点 `gauge` 计量
- 附着：`gauge = clamp(gauge + attach_amount, 0, max_gauge)`
- 第二种元素命中 → 触发反应，`gauge -= reaction_cost`；归零则状态消失
- **基础状态效果在 gauge > 0 期间始终生效**（关键：反应不会吃掉减速等效果，杜绝反协同）
- 同一敌人受内置冷却（ICD）限制，冷却期内不触发反应
- MVP 数值（全部走全局配置，禁止硬编码）：附着 2U / 消耗 1U / 上限 3U / 衰减 0 / ICD 0.5s

### 2.3 反应表（6 种，覆盖 6 种战术角色）

| 组合 | 反应 | 效果 | 战术角色 |
|------|------|------|---------|
| 火+冰 | 蒸汽爆破 | 范围 AoE 伤害 | 清杂 |
| 火+电 | 过载 | 爆炸 + 击退 | 位移控制 |
| 火+毒 | 燃爆 | 引燃毒雾，大范围持续燃烧 | 持续压制 |
| 冰+电 | 超导 | 冻结 + 向周围敌人传导 | 硬控扩散 |
| 冰+毒 | 脆化 | 易伤：受到伤害 +40% | 打 Boss |
| 电+毒 | 电解 | 麻痹：眩晕 1 秒 | 打断技能 |

**反应补偿铁律**：每个反应的收益必须 ≥ 被消耗元素量的价值（如超导用冻结补偿失去的冰量），保证玩家叠反应永不亏。

同元素重复命中 = 补充 gauge（叠层强化），不触发反应。

## 3. 内容系统概要

### 3.1 塔

- 每元素一条塔线：基础塔 → 3 级二选一分支
  - **附着型分支**：射速快、单发伤害低、附着量高（负责刷状态）
  - **触发型分支**：射速慢、单发伤害高（负责引爆）
- 中立塔（正式版）：**棱镜塔**（范围内反应效果 +50%）、**回响塔**（复读范围内最近一次反应）
- 构筑核心问题：附着方与触发方的**产耗节奏匹配**

### 3.2 敌人

1. 自带元素附着的敌人（如熔岩犬自带火）— 教学反应的最佳工具
2. 元素免疫/吸收敌人 — 逼阵容轮换
3. 净化者 — 周期性清除周围友军状态
4. 元素护盾 Boss — 血条分段，每段护盾只吃特定反应

### 3.3 经济

- **金币**：击杀获得，波次内建塔/升级
- **反应结晶**：仅由触发反应产出，跨关卡持久，用于科技树（正式版）

### 3.4 关卡

- 战役约 15 关：1–6 关每 2 关引入一种新元素及其反应，7–12 关引入复合敌人与中立塔，13–15 关为解题式关卡
- 地图多做**交叉路口与平行路径**，制造天然"反应位"
- 通关后开无尽模式（敌人抗性轮换）

## 4. 技术架构

### 4.1 最高铁律

1. **逻辑与数值彻底分离**：代码中不允许出现任何游戏数值字面量，所有数字住在 `res://data/` 下的 `.tres` 资源文件里。
2. **Resource 数据只读**：`.tres` 是共享引用，运行时永不修改其字段。所有运行时增益（buff、科技树）通过 `ModifierStack` 修饰层生效。
3. **系统间通信走 EventBus 信号**：UI、结算、成就等只订阅信号，互相零耦合。

### 4.2 目录结构

```
res://data/
  balance/global_config.tres      # GaugeConfig：全局默认值（MVP 唯一数值文件）
  elements/fire.tres  ice.tres  lightning.tres  poison.tres
  reactions/steam_burst.tres  overload.tres  superconduct.tres ...
  towers/fire_basic.tres  ice_basic.tres ...
  enemies/...
res://scripts/
  defs/          # Resource 类定义（纯数据结构：ElementDef / TowerDef / ReactionDef / EnemyDef / GaugeConfig）
  effects/       # ReactionEffect 及其子类（可组合积木）
  components/    # StatusComponent、HealthComponent 等挂在实体上的节点
  systems/       # ReactionSystem、EventBus、Balance（autoload）
res://scenes/
  towers/  enemies/  maps/  ui/
```

### 4.3 关键模式

**全局默认 + 局部覆盖**（MVP 最简、后期可扩）：

```gdscript
class_name GaugeConfig extends Resource
@export var default_attach: float = 2.0
@export var max_gauge: float = 3.0
@export var decay_per_sec: float = 0.0
@export var default_cost: float = 1.0
@export var reaction_icd: float = 0.5
```

`TowerDef.attach_override` / `ReactionDef.gauge_cost_override` 默认 `-1.0`（表示使用全局默认），需要差异化时局部覆盖，通过 `get_attach()` / `get_cost()` 查询方法解析。

**反应效果 = 可组合积木**（禁止 match 大分支）：

- 抽象基类 `ReactionEffect extends Resource`，接口 `apply(target, ctx: Dictionary)`
- 子类：`AoeDamageEffect` / `KnockbackEffect` / `StunEffect` / `StatModifierEffect`（易伤）/ `DotEffect` / `PropagateEffect`（包装另一个效果并扩散给周围敌人，用于超导传导）
- `ReactionDef.effects: Array[ReactionEffect]`，在 Inspector 里拖资源拼装
- 新增反应或修改反应效果 = 加数据，不改核心代码

**运行时核心流程**：

1. 弹丸命中 → 敌人 `StatusComponent` 收到 `(element, attach_amount, source_tower)`
2. 无状态或同元素 → 充能 gauge；异元素 → 调 `ReactionSystem.try_react()`
3. `ReactionSystem`（autoload）持有 `_table: Dictionary`（key = 两元素 id 排序拼接，保证"冰+火"="火+冰"），查表 → 校验 ICD → 扣 gauge → 遍历执行 `effects` → `EventBus.reaction_triggered.emit(def, enemy, source)`
4. `StatusComponent` 职责单一：gauge 的加/减/衰减/归零发 `status_expired` 信号，控制在 ~20 行

**塔的场景组合**：`Targeting` + `Weapon` + `ProjectileSpawner` 三个子节点，换元素只是换 `Weapon` 引用的 `TowerDef` 资源。

**反应归属**：反应伤害记给触发方的塔（source_tower），供击杀统计与塔成长使用。

### 4.4 平衡工作流（尽早搭建）

- headless 仿真场景：批量跑"标准波次 × 各塔组合"，输出各反应触发次数、DPS 占比到 CSV
- `.tres` ↔ CSV 双向同步脚本，支持在表格软件里批量调数
- 平衡流程 = 改数据 → 跑仿真 → 看报表，不改代码

## 5. MVP 范围（目标：2–3 周可玩原型）

**验证目标：触发反应的瞬间爽不爽、看不看得懂。**

包含：
- 4 座基础塔（无分支）+ 6 种反应 + Gauge 制（最简数值，全走 global_config）
- 1 张带交叉口的地图，10 波敌人（含 1 种自带火附着的怪）
- 单货币（金币）
- 敌人头顶状态图标 + 元素量环形进度条 + 反应飘字/特效占位

不包含（但架构必须预留）：塔分支、中立塔、科技树/结晶、衰减、差异化附着量、Boss、无尽模式。

### 建议实现顺序

1. 数据层骨架：`defs/` 全部 Resource 类 + `global_config.tres` + 4 元素 + 6 反应 `.tres`
2. `EventBus` + `ReactionSystem` + `StatusComponent`（先用单元/headless 测试验证 gauge 与反应表逻辑，不依赖画面）
3. 敌人（Path2D 寻路 + HealthComponent + StatusComponent）与波次生成器
4. 塔（Targeting/Weapon/ProjectileSpawner）与弹丸命中管线
5. 状态可视化 UI（图标 + gauge 环 + 反应飘字）
6. 地图、10 波配置、金币经济、胜负判定
7. headless 平衡仿真管线

---

## 6. 开发规范（Claude Code session 须遵守）

- 数值只进 `.tres`，代码出现魔法数字视为 bug
- 运行时修改 `.tres` 字段视为 bug（走 ModifierStack）
- 新反应/新效果通过新增 Resource 实现，禁止在 ReactionSystem 内写效果分支
- 所有跨系统通信经 EventBus，禁止系统间直接引用调用
- 每个核心系统（gauge、反应表、效果执行）先写 headless 可跑的测试再接画面
- GDScript 全程静态类型标注（`: float`、`-> void`），Resource 类必须 `class_name`
